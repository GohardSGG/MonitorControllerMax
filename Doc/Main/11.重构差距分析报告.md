# MonitorControllerMax 重构完成度差距分析报告

## 1. 概述

基于 `Doc/Main/8.未来架构演进路线.md` 与当前 `Source/` 目录代码的深度对比，我们发现虽然文件结构（Workspace 拆分）已经完成，但核心的架构升级目标（高性能、泛化、安全性）仍有部分未实现的工作。v2.8.0 已完成效果器音频处理、OSC 双向同步、Event Log UI 和音频安全加固。

以下是具体的差距分析：

## 2. 详细差距对比

### 2.1 基础设施与依赖管理 (Infrastructure)

| 规划目标 | 当前状态 | 差距 | 优先级 |
| :--- | :--- | :--- | :--- |
| **Git Submodules** | `Library/` 目录下仍是直接拷贝的代码 (Vendoring)。根目录无 `.gitmodules` 文件。 | **未实现**。无法方便同步上游 `nih-plug` 的更新。 | 高 |
| **资源内嵌** | Reactor 层 `web_assets.rs` 使用 `rust-embed` 内嵌 Web 前端资源。Slint GUI 使用 `slint::include_modules!()` 编译时内嵌 UI 资源。 | **已实现**。GUI 和 Web 资源均已内嵌到二进制文件中。 | ✅ 完成 |

### 2.2 实时安全性 (Real-time Purity)

| 规划目标 | 当前状态 | 差距 | 优先级 |
| :--- | :--- | :--- | :--- |
| **强制零分配 (Zero Alloc)** | `Plugin/Cargo.toml` 已启用 `assert_process_allocs`。`audio.rs` 代码经过了深度优化（栈上增益缓存、无分支计算、通道优先内存访问）。 | **已实现**。核心音频路径已达标。 | ✅ 完成 |
| **通信模型 (Triple Buffer)** | `Plugin` 持有 `Arc<InteractionManager>` (含 `RwLock`) 直接传递给 Audio 线程。未发现 `TripleBuffer` 结构体的引入。 | **未实现**。UI → Audio 仍然存在潜在的锁竞争风险（尽管可能是读锁），未达到"极致无锁"的目标。 | **极高** |
| **RingBuffer 事件流** | 目前主要依赖 `crossbeam::channel` 或直接原子状态读取。 | **未实现**。需引入 SPSC RingBuffer 以消除 Channel 的锁开销。 | 中 |

### 2.3 泛化与架构解耦 (Generalization)

| 规划目标 | 当前状态 | 差距 | 优先级 |
| :--- | :--- | :--- | :--- |
| **DspAlgorithm Trait** | `MonitorControllerMax` 仍直接调用 `audio::process_audio_with_layout`。业务逻辑与插件外壳深度耦合。 | **未实现**。无法复用目前的架构开发其他插件。代码未被抽象为通用模板。 | 高 |
| **InteractionManager 解耦** | `InteractionManager` 包含大量具体业务（Solo/Cut/Dim）。 | **未实现**。应抽象为泛型的状态机。 | 中 |

### 2.4 Web 交互完整性 (Web Completeness)

| 规划目标 | 当前状态 | 差距 | 优先级 |
| :--- | :--- | :--- | :--- |
| **Web 命令参数修改** | Solo/Mute/通道点击可工作。SetVolume, ToggleDim, ToggleCut 等 10 个命令标记为 `TODO: not applied`。 | **部分实现**。因 nih-plug 参数 `set_value()` 的 `pub(crate)` 可见性限制，Reactor 无法直接修改参数。需通过 pending queue 架构绕行。 | 高 |
| **WebSocket 状态推送** | `state_push_task()` 读取状态但未调用 `broadcast_tx.send()`。 | **未实现**。WebSocket 客户端无法收到实时状态更新。 | 高 |
| **通道名称映射** | `get_group_channels()` 使用 `"LS"`, `"RS"`, `"TFL"` 等名称，与 `STANDARD_CHANNEL_ORDER` 的 `"LSS"`, `"RSS"`, `"LTF"` 不匹配。 | **存在 Bug**。通道分组点击在部分布局下可能无效。 | 中 |
| **二进制协议 (Binary)** | 仍使用 `serde_json` 进行序列化通信。 | **未实现**。JSON 在高频数据传输（电平表）下会产生 GC 压力。 | 低 (长期) |

## 3. 总结与建议行动路线

目前代码处于 **v2.8.0 (结构重构 + GUI 迁移 + 效果器/安全加固完成)** 的稳定状态。编译通过，核心音频处理链路完整。

### 已完成的新功能 (v2.8.0)
- ✅ **效果器音频处理**: Low Boost (SUB ×1.5)、LFE +10dB (LFE ×3.16) 完整可用
- ✅ **OSC 双向同步**: 所有 6 个效果器按钮支持 UI → OSC 反馈发送，pending 机制防回路
- ✅ **Event Log UI**: logger → Slint `event-log-text` 属性绑定链路完整
- ✅ **音频安全加固**: 3 层静音默认值防护（master_gain / GainSmoothing / RenderState）

### 待实现功能
- 🟡 **High Boost**: UI 可用，音频处理逻辑缺失
- 🟡 **Mono Downmix**: OSC/UI 可用，音频 downmix 逻辑缺失
- 🟡 **SUB 通道 Solo 行为**: 当前豁免逻辑待用户确认

**建议立即执行的行动项（按优先级排序）：**

1. **[Critical] 实现 Triple Buffer**: 彻底切断 UI 线程与 Audio 线程的直接锁依赖。这是音频插件稳定性的基石。
2. **[Feature] Web 命令参数管线**: 设计 pending queue 架构，让 Reactor 的 Web 命令通过 `OscSharedState` 的 pending 标志位传递到 Editor/Plugin 层消费。
3. **[Feature] WebSocket 状态推送**: 完成 `state_push_task()` 的广播逻辑。
4. **[Bugfix] 通道名称映射**: 统一 `get_group_channels()` 与 `STANDARD_CHANNEL_ORDER` 的通道命名。
5. **[Refactor] 提炼 DspAlgorithm Trait**: 将具体业务逻辑剥离，验证架构的通用性。
6. **[Infra] 迁移至 Submodule**: 规范化依赖管理，准备长期维护。
